Core Capabilities, Skills, and Goals

这是一份放在核心仓库里的可视化说明：帮助维护者、测试 agent 和接入团队快速理解后端能力边界、协作技能、质量守卫和下一阶段目标。核心保持通用、外部配置驱动、本地优先；具体团队内容只应进入外部配置包或本地 Store。

This visual overview helps maintainers, testing agents, and adopting teams understand backend boundaries, collaboration skills, quality gates, and iteration goals. The core stays generic, externally configured, and local-first.

CoreCLI + Control plane + Store + Runner + Evidence + Reports

Profile-driven核心不携带团队场景，配置从外部发布。Team scenarios live in external profile bundles.

一条标准 agent 路径 / Standard Agent Flow

报告生成不应隐藏目标解析。agent 先发现候选，再用精确 ID 执行报告；失败用例进入报告，不阻断报告产出。Agents discover targets first, then run reports with exact IDs. Failed cases are report content, not report-generation failures.

Discover

发现可测目标
Find targets

用 discover 从 Store/catalog 搜索接口节点或工作流。Search interface nodes or workflows from Store/catalog.

Select

选择精确 ID
Select ID

从 JSON 结果中读取 id，后续 report 命令只接受 ID。Use returned IDs for report commands.

Execute

执行用例集合
Run tests

接口节点执行全部关联用例；工作流按绑定步骤执行。Run all attached cases or ordered workflow steps.

Record

写入本地索引
Record facts

运行、用例结果、证据摘要和后处理任务进入 Store。Persist run facts, Evidence indexes, and post-process tasks.

Render

生成临时报告
Render report

HTML 给人看，JSON 给机器读；失败原因保留在报告内。HTML for people, JSON for machines.

Review

审阅与回归
Review

报告、证据、日志和拓扑共同支撑后续分析。Reports, Evidence, logs, and topology support review.

后端能力总览 / Backend Capability Overview

当前后端不是单一 runner，而是一套本地控制平面：配置进入 Store，执行产生 Evidence，报告和页面再从同一份事实读取。The backend is a local control plane: profiles publish into Store, execution writes Evidence, and reports/pages read the same facts.

Store

本地事实库 / Local fact store

保存 schema、配置索引、运行记录、用例运行、证据索引、基线、耗时、日志、拓扑和后处理任务。Stores schema, catalogs, runs, Evidence indexes, logs, topology, timing, and tasks.

Profile

配置包生命周期 / Profile lifecycle

外部配置包可以创建、安装、打包、审计、验证、发布；运行产物和本地数据库会被过滤。Creates, installs, packs, audits, verifies, and publishes external bundles.

Read-model

目录视图 / Catalog views

把服务、工作流、接口节点、用例、模板、夹具、依赖和绑定转成页面与 agent 都能读取的视图。Builds UI and agent-readable views from profile assets.

后端数据流 / Backend Data Flow

这条链路解释为什么核心可以保持通用：团队差异进入配置包，运行事实进入 Store 和 Evidence，核心只负责通用编排。Team-specific differences enter profile bundles; runtime facts enter Store and Evidence; the core stays generic.

1. 外部配置
External profile

配置包保留在核心仓库外，作为可审阅源资产。Profiles stay outside core as reviewable source.

2. 审计发布
Audit and publish

profile audit/verify 检查引用完整性，再发布 read-model。Audit references, then publish read-models.

3. Store 索引
Store index

SQLite 保存目录、运行事实、基线、耗时和后处理任务。SQLite stores catalogs, run facts, baselines, timing, and tasks.

4. 目标发现
Discover

agent 或页面从 catalog 搜索接口节点和工作流。Agents/pages search interface nodes and workflows.

5. 执行记录
Execute

runner 写请求、响应、断言、摘要和 case run id。The runner records request, response, assertions, summary, and IDs.

6. 后处理
Post-process

拓扑、日志和重处理任务异步补齐，并记录状态和耗时。Topology/log work can complete asynchronously with timing records.

7. 报告审阅
Review

HTML 给人看，JSON 给机器读，失败项保留详情入口。HTML is human-readable; JSON is machine-readable.

技能说明 / Agent Skills

这里的“技能”指项目内推荐的 agent 工作方式，不是把团队场景写死在核心里。Skills here mean recommended agent workflows, not hardcoded team scenarios.

1

发现技能
Discovery

读取项目简介和命令帮助，调用 discover 搜索候选目标，再从结果中选择精确 ID。Read docs/help, discover targets, then select exact IDs.

2

报告技能
Reporting

使用精确 ID 生成 HTML/JSON 报告。报告是产物，用例失败是内容，不是 CLI 崩溃。Generate HTML/JSON reports; failed cases are report content.

3

配置技能
Configuration

维护外部配置包，先审计和验证，再发布到 Store/read-model。Maintain external profiles, audit and verify, then publish.

4

证据技能
Evidence

从 Store 和 evidence 文件定位请求、响应、运行摘要、日志与拓扑，形成可复查结论。Inspect requests, responses, summaries, logs, and topology.

5

守卫技能
Guardrails

每个后端切片都要有测试和守卫，避免通用核心被团队词汇或临时运行产物污染。Keep tests and guardrails around every backend slice.

命令面是 agent 和 CI 的首选入口；Control plane API 面向页面和长期服务模式。The CLI is the preferred entry for agents and CI; APIs support pages and service mode.

核心规则 / Core Rules

这些规则决定项目能否保持开源友好、跨团队可复用、可长期维护。These rules keep the project open-source friendly, cross-team reusable, and maintainable.

核心保持通用

核心代码、默认资产和文档不写入具体团队领域词。团队内容只能在外部配置包或本地运行数据中出现。

文件优先，数据库索引

配置包是可审阅源资产；Store 保存索引、read-model 和运行事实，不取代配置源。

本地优先

默认 SQLite 和本地 evidence，保证离线可跑、可复现、易调试。

报告不等于全绿

报告生成成功表示工具链可用；报告内的失败项表示被测系统或配置需要分析。

先发现，再执行

agent 必须先使用 discover 获取候选，再把精确 ID 传给 report。

小切片要可验证

行为变化先写测试，跑聚焦测试，再跑全量测试和守卫。

运行产物不提交

Evidence、日志、SQLite 数据库和临时报告属于运行态，不进入核心源码。

前后端共享 Store 视图

页面读取 Store/read-model，CLI 和 API 共同维护同一份运行事实。

迭代目标 / Iteration Goals

目标按“现在可交付、下一阶段强化、后续平台化”分层，避免把简单功能做复杂。Goals are layered into now, next, and later so simple capabilities stay simple.

Now

稳定 agent 验收闭环

固定 discover -> report 作为测试 agent 默认流程；确保接口节点全量报告、工作流报告、失败保留、HTML/JSON 输出稳定。

Next

强化前置数据与上下文传递

继续把 fixture、case dependency、workflow exports 和 Store 运行记录整合成通用前置数据能力，不写团队规则到核心。

Next

后处理异步化

拓扑、日志裁剪、报告补全等重处理任务落入 post-process task，记录耗时和状态，避免页面和 runner 被慢操作拖住。

Later

团队模式 Store

保持 SQLite 默认，补可选团队 Store 后端，让同一套 Store interface 支持本地和多人协作。

Later

公开发布准备

完善 profile 格式文档、报告样式、守卫说明、最小示例和贡献指南，让项目可被其他团队直接接入。
